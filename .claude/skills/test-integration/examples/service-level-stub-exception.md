# Service-level stub: approved exceptions

Infra-layer stubs are the default. Service-level stubs are allowed only when **both** conditions hold:

1. The service depends on infrastructure incompatible with a tmpdir (e.g. GitInfra requiring a real git repo)
2. The error condition under test cannot be injected through infra-level stubs

## Affected files

### run.integration.test.ts

`PipelineService` stubbed via `makePipelineServiceThrowingStub`:

- `PipelineMaxRetriesError` is generated by internal pipeline retry logic and cannot be simply injected through `claudeCodeInfra`
- `AbortService` abort-state tests also require service-level injection to reproduce mid-pipeline signal state

```ts
function makePipelineServiceThrowingStub(err: Error): PipelineService {
  return {
    run: vi.fn(async function* (): AsyncGenerator<PipelineTaskResult> {
      yield* [] as PipelineTaskResult[]
      throw err
    })
  } as unknown as PipelineService
}

setupContainer({
  config: buildTestConfig(dir),
  infras: { gitInfra: buildGitInfraStub(), fileMoveInfra: buildFileMoveInfraStub() },
  services: { pipelineService: makePipelineServiceThrowingStub(new PipelineMaxRetriesError(0, 3)) }
})
```

### run.task-output.integration.test.ts

`PipelineService` stubbed via `makePipelineServiceYieldingStub`:

- `retry` / `pipeline_end` `PipelineTaskResult` variants are too complex to produce via infra stubs alone
- `onStreamEvent` callback testing requires the same approach

### inspect.integration.test.ts

`PipelineFileService` only:

- `PipelineFileService` depends on `GitInfra` operations (`getDiffStat`, `getHead`, `moveToDone`, `commitMove`) that fail in a non-git tmpdir

## Before applying this pattern

1. First confirm infra stubs cannot achieve the same result
2. Verify both conditions above are met
3. Leave a comment in the test file explaining why (see the comment above `makePipelineServiceThrowingStub` in existing files)
