# Planning Pipeline: Three-Agent Architecture

**Type:** Discovery

## Context

Applies when implementing, modifying, or debugging the planning workflow. The planning system
uses three agents in sequence, each reading and producing specific files under `plans/<slug>/`.

## What happened / What is true

Planning is realised as a three-agent pipeline:

```
meta-plan/plan  →  code-base-survey/survey  →  meta-pipeline-creator/create
```

### Agent roles and I/O

| Agent | Procedure | Input | Output |
|-------|-----------|-------|--------|
| meta-plan/plan | `plan.md` | feature description | `plans/<slug>/` directory |
| code-base-survey/survey | `survey.md` | `plans/<slug>/layers.md` + `<layer>.md` | `plans/<slug>/gotchas.md`, corrected layer files |
| meta-pipeline-creator/create | `create.md` | `plans/<slug>/layers.md` + `<layer>.md` + `gotchas.md` | `pipelines/implement__<slug>.json` |

### Directory structure under `plans/<slug>/`

```
plans/<slug>/
  brief.md      Goal and design decisions (generated by meta-plan)
  layers.md     Layer manifest — routing input for meta-pipeline-creator/create
  <layer>.md    Per-layer specification (skeleton by meta-plan, refined by code-base-survey)
  gotchas.md    Warnings and reuse candidates (generated by code-base-survey)
```

### Core design principle

**Interface design is the primary goal.** Each `<layer>.md` must lead with interface
definitions (type signatures, port methods). Code sketches follow. The correctness of these
interfaces determines the success of the subsequent multi-agent implementation run.

`layers.md` acts as a routing manifest for `meta-pipeline-creator/create`, listing which layer
files exist and must be processed. Implementation order is fixed:

```
errors → types → infrastructures → repositories → domains → services → validators → cli
```

### Why three agents instead of one

The previous single-agent approach (`plan-feature.md`, now renamed `plan.md`) had one
agent perform survey, design, and pipeline generation, writing all output to a single file.
The new split allows each agent to read only the files it needs, reduces context window
pressure, and makes each stage independently re-runnable.

## Do

- Run agents in the sequence: `meta-plan/plan` → `code-base-survey/survey` → `meta-pipeline-creator/create`
- Put interface definitions at the top of each `<layer>.md` before any code sketches
- Ensure `layers.md` lists all layer files before running `meta-pipeline-creator/create`

## Don't

- Don't skip `code-base-survey/survey` — it produces `gotchas.md` that `meta-pipeline-creator/create` depends on
- Don't change the implementation order in `layers.md` away from the canonical sequence above
